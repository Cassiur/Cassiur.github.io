模块三 阅读计数的双写策略

3.1 每次打开文章，系统做三件事

  1. 把文章内容查出来给你看 (查 DB)
  2. 文章阅读量 +1 (写 DB + 写 Redis)
  3. 作者的阅读量也 +1 (写 Redis)

3.2 核心代码

// ArticleReadServiceImpl.queryFullArticleInfo() 中调用：
countService.incrArticleReadCount(article.getAuthor(), articleId);

// CountServiceImpl.incrArticleReadCount() 的实现：
public void incrArticleReadCount(Long authorUserId, Long articleId) {

    // ① 直接写 DB：UPDATE article SET read_count = read_count + 1
    articleDao.incrReadCount(articleId);

    // ② Redis Pipeline：一次网络往返，执行两个命令
    RedisClient.pipelineAction()
        .add("article_statistic_" + articleId, "readCount",
            (conn, key, val) -> conn.hIncrBy(key, val, 1))
        .add("user_statistic_" + authorUserId, "readCount",
            (conn, key, val) -> conn.hIncrBy(key, val, 1))
        .execute();
}

3.3 Pipeline 为什么能减少网络往返？

不用 Pipeline（2次网络往返）：

客户端 → Redis: HINCRBY key1 field1 1  (RTT: 1ms)
Redis → 客户端: OK
客户端 → Redis: HINCRBY key2 field2 1  (RTT: 1ms)
Redis → 客户端: OK
总耗时 ≈ 2ms

用 Pipeline（1次网络往返）：

客户端 → Redis: [HINCRBY key1 field1 1,
                HINCRBY key2 field2 1]  (RTT: 1ms)
Redis → 客户端: [OK, OK]
总耗时 ≈ 1ms
类比： 像点外卖——一次点两个菜比分开点两次快。Pipeline 把多个命令打包成一个 TCP 数据包发送，Redis 顺序执行后一次性返回所有结果。

3.4 为什么用 Hash 而不是 6 个 String Key？

Redis 中的数据是这样的：

Key: devlink_article_statistic_123
Hash内部:
  praiseCount      → 42    (点赞数)
  collectionCount  → 15    (收藏数)
  commentCount     → 8     (评论数)
  readCount        → 1024  (阅读数)

Key: devlink_user_statistic_456
Hash内部:
  followCount      → 100   (关注数)
  fansCount        → 200   (粉丝数)
  articleCount     → 35    (文章数)
  praiseCount      → 500   (获赞数)
  collectionCount  → 80    (被收藏数)
  readCount        → 5000  (被阅读数)
方案内存占用一次查全部原子自增
6个 String高（每个Key有独立开销）需要6次网络请求INCR 原子
1个 Hash低（小Hash用压缩编码）1次 HGETALLHINCRBY 原子

3.5 Pipeline vs 事务 vs Lua 脚本

特性PipelineMULTI/EXEC (事务)Lua Script
网络往返1次1次1次
原子性❌ 无✅ 有（但不回滚）✅ 有
能否用上一条结果❌ 不可以❌ 不可以✅ 可以
你的场景够用 ✓过度过度

模块四 Caffeine 本地缓存

4.1 侧边栏里有什么？

打开 DevLink 文章详情页，右侧有这些内容：📢 公告通知、🔥 热门文章、🏆 月度活跃排行榜、📚 优质PDF资源、📖 相关推荐文章。

这些内容不是每秒钟都变的——公告可能一周才改一次，热文可能几分钟更新一次。

4.2 缓存层级

用户请求侧边栏
    │
    ├─ L1: Caffeine 本地缓存 (JVM内存, 微秒级)
    │     └─ 命中 → 直接返回，不走网络
    │
    ├─ L2: Redis 远程缓存 (网络IO, 毫秒级)
    │     └─ 侧边栏场景不走这层，但排行榜走了
    │
    └─ L3: MySQL 数据库 (磁盘IO, 几十毫秒)

4.3 Caffeine 配置

@Bean("caffeineCacheManager")
public CacheManager cacheManager() {
    CaffeineCacheManager cacheManager = new CaffeineCacheManager();
    cacheManager.setCaffeine(Caffeine.newBuilder()
        .expireAfterWrite(5, TimeUnit.MINUTES)  // 写入后5分钟过期
        .initialCapacity(100)                    // 初始100条
        .maximumSize(200)                        // 最多200条
    );
    return cacheManager;
}

4.4 @Cacheable 的工作流程

调用方 → CGLIB代理 → CacheInterceptor.before():
    Caffeine.get("homeSidebar")
      ├─ 有 → 直接返回，不执行方法体 ← 这就是"缓存命中"
      └─ 无 → 执行真实方法体(查DB、查Redis)
              → CacheInterceptor.after():
                  Caffeine.put("homeSidebar", result)  ← "写入缓存"

4.5 ⚠️ 重要陷阱：内部调用导致缓存失效

// ❌ 错误写法：this.pdfSideBar() → 不会走缓存！
// ✅ 正确写法：必须通过代理对象调用
list.add(SpringUtil.getBean(SidebarServiceImpl.class).pdfSideBar());
原因： @Cacheable 是 Spring AOP 实现的。AOP 的拦截逻辑在代理对象上。直接 this.xxx() 调用的是原始对象，跳过了代理，缓存拦截器不会执行。

模块五 点赞/收藏的 UserFoot 设计

5.1 表设计：一条记录存所有操作

CREATE TABLE user_foot (
    id              BIGINT PRIMARY KEY,
    user_id         BIGINT,     -- 谁操作的
    document_id     BIGINT,     -- 操作的文章/评论ID
    document_type   INT,        -- 0=文章, 1=评论
    document_user_id BIGINT,    -- 文档作者ID
    praise_stat     INT,        -- 0=未点赞, 1=已点赞
    collection_stat INT,        -- 0=未收藏, 1=已收藏
    comment_stat    INT,        -- 0=未评论, 1=已评论
    read_stat       INT,        -- 0=未读, 1=已读
);

-- 唯一索引：一个用户对一篇文档只有一条记录
CREATE UNIQUE INDEX uk_user_document
    ON user_foot(user_id, document_id, document_type);

设计思路：一条记录 = 一个用户对一个文档的全部互动状态，避免了"点赞一张表、收藏一张表、阅读一张表"的多表 JOIN。

5.2 点赞流程

① SELECT → 查是否有足迹记录
② 不存在 → INSERT 新记录(praise_stat=1)
   存在且未点赞 → UPDATE praise_stat=1
   已点赞 → 幂等返回(或取消→设为0)
③ 发送 RabbitMQ 消息 → 异步更新计数/积分/通知

5.3 并发问题

代码中标注了 FIXME：在高并发下可能出现查询数据与DB不一致。比如快速双击"赞→取消"，两条线程同时读到 praiseStat=0，导致操作丢失。

可能的解决方案：分布式锁、事务+悲观锁、CAS乐观锁。但作者选择不处理——因为足迹数据的准确性影响不大，不值得引入复杂性。这是工程上务实的取舍。

6.2 ZSet 排行榜原理

Key: activity_rank_20260607 (日榜)
┌──────────┬────────┐
│  Member  │ Score  │

模块七 主从读写分离

7.1 为什么需要读写分离？

读请求占 90% 以上，一台 MySQL 扛不住。用主从架构：Master DB 处理写，Slave DB 分担读。

写请求 ──→ Master DB (主库)
              │ (binlog同步)
    ┌─────────┼─────────┐
    ▼         ▼         ▼
 Slave 1  Slave 2  Slave 3
    ↑         ↑         ↑
  用户读    用户读    用户读

7.2 实现：AOP + ThreadLocal + AbstractRoutingDataSource

// ① AOP 切面：拦截 @DsAno 注解
@Around("@annotation(DsAno)")
public Object around(ProceedingJoinPoint pjp) {
    DsContextHolder.set("SLAVE");  // ThreadLocal 设值
    try { return pjp.proceed(); }
    finally { DsContextHolder.reset(); }
}

// ② MyBatis 获取连接时调用：
protected Object determineCurrentLookupKey() {
    return DsContextHolder.get();  // "SLAVE" → 连从库
}
为什么用 ThreadLocal？ 每个 HTTP 请求由独立线程处理，ThreadLocal 保证同一个请求线程内数据源选择一致，且不同请求互不干扰。

模块九 Caffeine vs Redis 对比

维度CaffeineRedis
位置JVM 堆内存远程服务器
速度微秒级毫秒级（网络开销）
容量MB 级（受JVM堆限制）GB 级

项目中侧边栏用 Caffeine 的原因：数据变化不频繁、读多写少、微秒级响应、短暂不一致可接受。

Caffeine W-TinyLFU vs 传统 LRU

传统 LRU 的问题：扫描性操作会把热点数据挤出缓存。比如有人用脚本遍历所有文章，LRU 会把热门文章挤出。

W-TinyLFU 不仅看"最近"，还看"频率"——热门文章访问频率高，即使暂时没被访问也不会被挤出。三层结构：Window Cache → Main Cache → TinyLFU Filter。
