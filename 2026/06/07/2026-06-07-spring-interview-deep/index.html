Q37：拦截器（Interceptor）和过滤器（Filter）的区别？

一句话总结：Filter 是 Servlet 规范，拦截所有 HTTP 请求；Interceptor 是 Spring MVC 规范，只拦截进入 Controller 的请求。执行顺序是：Filter → Interceptor → Controller。

深度解析：

想象你去办证件：

  • Filter（过滤器）：像大门保安，所有进大楼的人（所有 HTTP 请求）都要经过他检查。
  • Interceptor（拦截器）：像楼层前台，只有去特定楼层（Controller）的人才会被他检查。

执行顺序图：

对比项 Filter Interceptor
规范 Servlet 规范（Java EE） Spring MVC 规范
拦截范围 所有 HTTP 请求（包括静态资源） 只拦截进入 Controller 的请求
注入 Spring Bean 不能直接注入（需用 WebApplicationContext 获取） 可以直接注入（是 Spring 管理的 Bean）
执行时机 最外层（最早执行，最晚结束） DispatcherServlet 内部（Controller 前后）
配置方式 @WebFilter 或 web.xml 实现 WebMvcConfigurer 的 addInterceptors

面试加分回答：

Filter 和 Interceptor 的核心区别是规范不同、拦截范围不同。Filter 是 Servlet 规范，拦截所有请求；Interceptor 是 Spring MVC 规范，只拦截 Controller 请求。实际项目中，登录校验、权限校验用 Interceptor 更方便（可以直接注入 TokenService 等 Bean），而编码转换、CORS 处理用 Filter 更合适（在 Spring 之前执行）。

Q39：Spring 中的 @Value 和 @ConfigurationProperties 有什么区别？

一句话总结：@Value 适合注入单个属性，@ConfigurationProperties 适合批量绑定属性（类型安全、支持校验、支持复杂类型）。

深度解析：

方式一：@Value（注入单个属性）

方式二：@ConfigurationProperties（批量绑定，推荐）

对比：

对比项 @Value @ConfigurationProperties
绑定方式 逐个绑定（@Value(“${key}”)） 批量绑定（prefix + 字段名自动匹配）
类型安全 ❌ 不支持（key 写错编译不报错） ✅ 支持（字段名和配置文件自动匹配）
复杂类型 ❌ 不支持（List、Map 需自定义转换器） ✅ 支持（List、Map、嵌套对象）
数据校验 ❌ 不支持 ✅ 支持（@Validated + @NotNull 等）
默认值 支持（${key:defaultValue}） 不支持（字段类型为包装类，未配置即为 null）
推荐场景 注入少量零散配置 绑定一组相关配置（如数据库连接池配置）

面试加分回答：

@ConfigurationProperties 的核心优势是类型安全和支持复杂类型。实际项目中，所有的自定义配置都应该用 @ConfigurationProperties（创建一个 XxxProperties 类），所有的零散配置（如 @Value(“${app.version}”)）才用 @Value。另外，@ConfigurationProperties 支持 JSR-303 校验（@Validated + @NotNull），可以在应用启动时就能发现配置错误，而不是运行时才报错。

Q41：Spring 中 WebClient 和 RestTemplate 的区别？

一句话总结：RestTemplate 是同步阻塞的（已不推荐），WebClient 是异步非阻塞的（基于 Project Reactor，支持响应式编程），是 Spring 5 引入的新一代 HTTP 客户端。

深度解析：

对比：

对比项 RestTemplate WebClient
编程模型 同步阻塞（调用线程会等待） 异步非阻塞（基于 Reactor）
性能 低（线程阻塞，吞吐量受限） 高（少量线程处理大量请求）
响应式支持 ❌ 不支持 ✅ 支持（Mono、Flux）
超时配置 复杂（需自定义 ClientHttpRequestFactory） 简单（.timeout(Duration))
状态 deprecated（Spring 5+ 不推荐） 推荐（Spring 5+ 的官方推荐）
适用场景 传统 MVC 项目（同步调用） 响应式项目（Spring WebFlux）或高并发场景

面试加分回答：

RestTemplate 被 deprecated 的根本原因是同步阻塞模型的性能瓶颈：一个请求对应一个线程，线程在等待响应时会一直阻塞，导致线程资源耗尽。WebClient 基于 Reactor，采用事件驱动模型，少量线程就能处理大量并发请求，吞吐量远超 RestTemplate。不过，如果你的项目不是响应式项目（没用 Spring WebFlux），用 RestTemplate 也完全没问题，只是未来可能会被移除。新项目建议直接用 WebClient。

Q43：Spring 中的事务传播机制有哪些？什么场景用哪一类？

一句话总结：Spring 定义了 7 种事务传播机制，决定当一个事务方法调用另一个事务方法时，事务如何传播。最常用的是 REQUIRED（默认，支持当前事务）和 REQUIRES_NEW（新建事务，挂起当前事务）。

深度解析：

7 种传播机制：

传播机制 说明 场景
REQUIRED（默认） 支持当前事务，如果当前没有事务，就新建一个 大部分场景（如 Service 方法调用另一个 Service 方法）
SUPPORTS 支持当前事务，如果当前没有事务，就以非事务方式执行 查询方法（可有可无事务）
MANDATORY 支持当前事务，如果当前没有事务，就抛异常 强制要求调用方有事务（不常用）
REQUIRES_NEW 新建事务，如果当前有事务，把当前事务挂起 日志记录（无论主事务是否回滚，日志都要提交）
NOT_SUPPORTED 以非事务方式执行，如果当前有事务，把当前事务挂起 不希望事务影响性能的场景（如发送短信）
NEVER 以非事务方式执行，如果当前有事务，就抛异常 强制要求非事务（不常用）
NESTED 如果当前有事务，就在嵌套事务中执行（保存点） 嵌套事务（外层回滚，内层也回滚；内层回滚，外层可以继续）

示例：REQUIRED vs REQUIRES_NEW：

嵌套事务（NESTED）示例：

面试加分回答：

事务传播机制是 Spring 事务面试的超高频题。最核心的是 REQUIRED（默认，90% 场景用这个）和 REQUIRES_NEW（独立事务，如日志记录）。NESTED 和 REQUIRES_NEW 的区别是：REQUIRES_NEW 是完全独立的两个事务，外层回滚不影响内层；NESTED 是嵌套事务（保存点），外层回滚内层也回滚，但内层回滚不影响外层。实际项目中，如果一个方法需要调用另一个事务方法，且希望被调用方法的回滚不影响当前方法，就用 REQUIRES_NEW；如果希望被调用方回滚不影响调用方，但调用方回滚时被调用方也回滚，就用 NESTED。

Q44：Spring 中事务的隔离级别有哪些？和数据库隔离级别的关系？

一句话总结：Spring 定义了 5 种事务隔离级别（对应数据库的 4 种 + 默认使用数据库隔离级别），通过 @Transactional(isolation = ...) 指定，最终由数据库实现，Spring 只是传递参数。

深度解析：

Spring 的 5 种隔离级别：

Spring 隔离级别 对应数据库 说明
DEFAULT（默认） 使用底层数据库的默认隔离级别 MySQL 默认 REPEATABLE_READ，Oracle 默认 READ_COMMITTED
READ_UNCOMMITTED 读未提交 脏读、不可重复读、幻读都可能发生（几乎不用）
READ_COMMITTED 读已提交 解决脏读，但不可重复读和幻读仍可能发生
REPEATABLE_READ 可重复读 解决脏读和不可重复读，但幻读仍可能发生
SERIALIZABLE 串行化 所有问题都解决，但性能极差（表级锁）

重要：Spring 本身不实现隔离级别，它只是通过 JDBC 把隔离级别传递给数据库，由数据库来实现。如果数据库不支持某个隔离级别（如 MySQL 支持所有 4 种，但 Oracle 只支持 READ_COMMITTED 和 SERIALIZABLE），设置无效。

面试加分回答：

Spring 的隔离级别最终是由数据库实现的，Spring 只是传参。很多人以为设置了 @Transactional(isolation = ...) 就万事大吉，但实际上：① 如果数据库不支持该隔离级别（如 Oracle 不支持 REPEATABLE_READ），设置无效；② 隔离级别越高，性能越差（SERIALIZABLE 几乎等于单线程执行）。实际项目中，MySQL 默认用 REPEATABLE_READ 就够了，只有特殊场景（如金融转账）才需要手动设置更高隔离级别。另外，Spring 的 DEFAULT 隔离级别表示”使用数据库的默认隔离级别”，这是最常用、最安全的设置。

@Transactional 失效是 Spring 面试的超高频题。90% 的失效都是因为自调用（同一个类中方法互调）。原因是 Spring 的事务是通过代理对象实现的，自调用时 this 指向目标对象，而不是代理对象，所以事务逻辑不会生效。解决办法是拆类（将两个方法放到不同的 Service 中）。另外，很多人不知道 @Transactional 默认只回滚 RuntimeException 和 Error，如果遇到 IOException 或 SQLException 这些受检异常，事务不会回滚，必须手动指定 rollbackFor = Exception.class。

Q47：Spring 中单例 Bean 是线程安全的吗？

一句话总结：单例 Bean 本身不是线程安全的，因为所有线程共享同一个 Bean 实例。但是，如果 Bean 没有成员变量（或无状态），就是线程安全的；如果有成员变量（或有状态），就需要加锁或使用 ThreadLocal。

Spring 中没有”线程安全的 Bean”这种概念，只有”线程安全的代码”。单例 Bean 是否线程安全，取决于Bean 是否有共享状态（成员变量）。实际项目中，大部分的 Service、Dao 都是无状态的（只有方法，没有成员变量），所以是线程安全的，不需要担心。如果你必须在单例 Bean 中保存状态（如计数器），要么用 ThreadLocal（每个线程独享），要么用 AtomicInteger（CAS 无锁并发），要么改为原型作用域（@Scope("prototype")）。

@Lazy 的核心价值是解决循环依赖和加快启动速度。循环依赖的最佳解决方案是重构代码（打破循环），但如果实在无法避免，@Lazy 是最简单的临时方案。另外，对于启动时间敏感的应用（如 FaaS、Serverless），可以用 @Lazy 标注所有非关键的 Bean，让应用快速启动，只有在真正需要时才创建这些 Bean，缩短启动时间。但要注意：@Lazy 只是把 Bean 的创建推迟到第一次使用，并不会减少最终的内存占用。

  2. Bean 生命周期 + 循环依赖（三级缓存，面试超高频）
  3. AOP（动态代理、通知类型、切面顺序）
  4. 事务管理（传播机制、隔离级别、事务失效场景）
  5. Spring Boot 自动配置原理（@SpringBootApplication、条件注解）
  6. Spring MVC 工作流程（DispatcherServlet、HandlerMapping）
  7. 高频面试题（设计模式、事件机制、全局异常处理）
