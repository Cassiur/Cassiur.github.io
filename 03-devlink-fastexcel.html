模块一 为什么要并发导出？

1.1 场景：导出 100 万条数据到 Excel

❌ 不用并发（慢在哪里）？ ① 查 DB：SELECT * FROM request_count（3 秒） → 100 万条数据从磁盘加载到内存，慢在 IO 等待 ② 写 Excel：FastExcel.write(data)（0.5 秒） → 写磁盘，相对较快 总耗时 ≈ 3.5 秒 （用户点击"导出"后，要等 3.5 秒才能下载文件） ✅ 用并发（快在哪里）？ ① 拆成 10 个分片，10 个线程 同时查 DB（0.4 秒） → 原来 1 个线程查 3 秒，现在 10 个线程并行查，每个查 10 万条 → 理想情况：3秒 / 10 = 0.3秒，加上线程调度开销 ≈ 0.4秒 ② 等所有分片查完，FastExcel 一次性写（0.5 秒） → 写 Excel 不是瓶颈，单线程写就够 总耗时 ≈ 1.4 秒（提升 2.5 倍！）
✅ 核心思想：查 DB 是瓶颈（IO 密集），用多线程并发查； 写 Excel 不是瓶颈（CPU + 磁盘 IO），单线程写就够，而且必须单线程（线程安全）。

1.2 为什么写 Excel 要单线程？

Q：FastExcel 的写操作为什么不能并发？

FastExcel（和 EasyExcel）的 ExcelWriter 不是线程安全的：

  • 内部有 currentSheet（当前 Sheet）、rowCount（当前行号）等共享状态
  • 多线程同时调 write() → 行号错乱、Sheet 错乱、文件损坏
  • 必须单线程写，或者每个线程写独立文件最后合并（更复杂，不值得）

项目的做法：多线程只负责"查数据"，全部查完后由主线程"写 Excel"。

1.3 性能对比：不同数据量的耗时

数据量不用并发用并发（10 线程）提升倍数
10 万条≈ 0.4 秒≈ 0.2 秒2 倍
100 万条≈ 3.5 秒≈ 1.4 秒2.5 倍
500 万条≈ 18 秒≈ 6 秒3 倍
1000 万条≈ 35 秒（可能 OOM）≈ 10 秒（可能 OOM）3.5 倍
⚠️ 注意：并发查 DB 只能加速"查数据"阶段。 如果数据量太大（1000 万条），即使并发查，内存存不下也会 OOM。 这种情况要用"生产者-消费者模式"（后面详细讲）。

模块二 线程池核心概念（面试必背）

2.1 为什么要用线程池？（不是直接 new Thread）

不用线程池（坏做法）： for (int i = 0; i < 100; i++) { new Thread(() -> queryData()).start(); // 每次都创建新线程 } → 创建 100 个线程，每个线程占 1MB 栈内存 = 100MB 内存浪费 → 线程创建/销毁开销大（每次 ~1ms） → 系统可能撑不住（Linux 默认线程数限制 ~1000） 用线程池（好做法）： 线程池里只有 8 个核心线程（和 CPU 核心数对齐） → 100 个任务来了，8 个线程轮流处理，不创建新线程 → 节省内存（8 个线程 × 1MB = 8MB） → 节省时间（不用每次创建/销毁线程）

2.2 线程池的 7 个参数（面试必背！）

// Java 线程池的核心构造器（必须会写！）
new ThreadPoolExecutor(
    int corePoolSize,      // ① 核心线程数（长期驻留的线程数）
    int maximumPoolSize,   // ② 最大线程数（忙时最多能开多少线程）
    long keepAliveTime,      // ③ 空闲线程存活时间
    TimeUnit unit,            // ④ 时间单位
    BlockingQueue<Runnable> workQueue,  // ⑤ 工作队列（存等待执行的任务）
    ThreadFactory threadFactory,                 // ⑥ 线程工厂（给线程起名字）
    RejectedExecutionHandler handler               // ⑦ 拒绝策略（队列满了怎么办）
)
参数项目中设置说明面试追问
corePoolSize8核心线程数（长期驻留）为什么是 8？（CPU 核心数）
maximumPoolSize16最大线程数为什么是 16 不是 80？
keepAliveTime60 秒非核心线程空闲 60 秒后回收为什么是 60 秒？
workQueueArrayBlockingQueue(11)有界队列（最多 11 个任务在排队）为什么不用 LinkedBlockingQueue？
handlerCallerRunsPolicy调用者线程自己跑任务（背压！）为什么不用 AbortPolicy？

2.3 线程池执行流程（面试必画！）

任务提交的完整流程： ① 任务来了 → 如果当前线程数 < corePoolSize → 创建新线程（核心线程）来执行这个任务 ② 任务继续来 → 核心线程都在忙 → 任务放进 工作队列（workQueue） ③ 任务继续来 → 工作队列满了 → 如果当前线程数 < maximumPoolSize → 创建新线程（非核心线程）来执行任务 ④ 任务继续来 → 线程数已达 maximumPoolSize，队列也满了 → 触发 拒绝策略（handler） 关键：项目里 maximumPoolSize=16，但队列只有 11 个位置 → 8 个核心线程在忙 + 11 个任务在排队 = 19 个任务 → 第 20 个任务来时，才会创建非核心线程（到第 16 个） → 但实际上：队列满了（11 个）就会触发 CallerRunsPolicy → 不会创建非核心线程！（因为 CallerRunsPolicy 会阻塞提交）

2.4 为什么 corePoolSize = 8？（CPU 核心数）

Q：导出是 IO 密集型，为什么 corePoolSize 不是 CPU×2？

答：这取决于"查 DB"是 IO 密集还是 CPU 密集。

  • 纯 IO 密集（查 DB、调 RPC）：线程大部分时间在等（等 DB 返回），CPU 是空闲的 → 应该设 CPU核心数 × 2（让更多线程同时等，提高吞吐量）
  • IO + 计算混合（查 DB + 数据转换）：线程有一部分时间在用 CPU → 应该设 CPU核心数 + 1 或 CPU核心数 × 1.5
  • 项目的情况：查 DB 是 IO 密集，但每个任务查完数据后还要做数据转换（DTO 转换）→ IO + 计算混合 → 设 8（CPU 核心数）是合理的，如果想更激进可以设 12~16

面试加分："我们的场景是 IO + 计算混合，所以设了 CPU 核心数。如果是纯 IO 场景（比如只是转发请求），会设 CPU×2。"

long total = requestCountService.count();  // 先查总记录数

3.2 第 2 步：创建自定义线程池（关键配置！）

// 文件位置：同上（第 60-66 行左右）
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    CORE_POOL,        // 8   → 核心线程数（和 CPU 核心数对齐）
    MAX_POOL,         // 16  → 最大线程数
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(shardCount + 1),  // ★ 有界队列！（关键！）
    r -> new Thread(r, "excel-export-" + r.hashCode()),
    new ThreadPoolExecutor.CallerRunsPolicy()   // ★ 背压策略！（关键！）
);

// ★ 为什么用 ArrayBlockingQueue（有界队列）？
//   → 防止任务无限堆积，导致 OOM
//   → 队列大小 = shardCount + 1（刚好够用，不浪费内存）

// ★ 为什么用 CallerRunsPolicy（调用者运行策略）？
//   → 队列满了，提交任务的线程（主线程）自己来跑这个任务
//   → 主线程被占用 → 不能继续提交新任务 → 天然背压（限速）！

3.3 为什么用 ArrayBlockingQueue（有界队列）？

队列类型是否有界可能的问题项目选择
LinkedBlockingQueue（无界）❌ 无界（默认 Integer.MAX_VALUE）任务无限堆积 → OOM❌ 不用
ArrayBlockingQueue（有界）✅ 有界（构造时指定大小）队列满了 → 触发 拒绝策略✅ 项目用的这个
SynchronousQueue（同步）✅ 容量为 0（不存任务）直接交付给线程，适合maximumPoolSize 很大的场景❌ 不适合（分片数固定）
⚠️ OOM 风险：如果 100 万数据拆成 1000 个分片， shardCount + 1 = 1001，任务会全部放进队列（1001 个）， 每个任务查 1000 条数据（存到内存），可能撑爆内存。
优化方案：限制最大分片数（比如最多 20 个分片），超过的部分串行处理。 或者用"生产者-消费者模式"（不存全量数据到内存）。

3.4 CallerRunsPolicy（背压策略）是什么？

场景：队列满了（10 个任务在排队），再来新任务怎么办？ ① AbortPolicy（默认）： 抛 RejectedExecutionException → 任务丢失 （不适合导出场景：用户点击"导出"，结果任务被丢弃，用户不知道为什么） ② DiscardPolicy（丢弃）： 直接丢弃新任务 → 任务丢失（和上面一样不适合） ③ CallerRunsPolicy（调用者运行，项目用的）： → 提交任务的线程（主线程）自己来跑这个任务 → 主线程被占用 → 不能继续提交新任务 → 天然背压（限速）！ → 不会丢任务，也不会 OOM（因为提交速度自动慢下来） ④ DiscardOldestPolicy（丢弃最老的）： 丢弃队列里最老的任务，放入新任务 （适合"最新数据最重要"的场景，不适合导出）
✅ CallerRunsPolicy 的妙处：不丢任务 + 自动限速（背压）。 主线程被占用时，提交速度自然慢下来，不会压垮系统。
生活比喻：餐厅厨房满了（队列满），服务员（主线程）自己去帮忙做菜（执行任务）， 就不能继续接新订单了 → 顾客（任务提交方）看到服务员在忙，就会等一会再点餐 → 天然限速！

3.5 第 3 步：提交分片任务 + CountDownLatch 等待

// 文件位置：同上（第 68-85 行左右）
CountDownLatch latch = new CountDownLatch(shardCount);
List<List<RequestCountDO>> result = new ArrayList<>();

for (int i = 0; i < shardCount; i++) {
    final int shardIndex = i;
    executor.submit(() -> {
        try {
            // 每个线程查自己的分片（OFFSET = shardIndex * BATCH_SIZE）
            List<RequestCountDO> shardData =
                requestCountService.queryByPage(shardIndex * BATCH_SIZE, BATCH_SIZE);

            // ★ 需要同步！因为 result 是共享的 ArrayList
            synchronized (result) {
                result.addAll(shardData);  // 把分片数据合并到总结果
            }
        } finally {
            latch.countDown();  // 完成一个分片，计数器 -1
        }
    });
}

latch.await();  // ★ 主线程等待所有分片完成（计数器到 0）
// 到这里，result 已经包含所有分片的数据 → 可以写 Excel 了

3.6 第 4 步：单线程写 Excel

// 所有分片数据已就绪，主线程写 Excel（单线程，线程安全！）
FastExcel.write(response.getOutputStream(), RequestCountDO.class)
    .sheet("request_count")
    .doWrite(result);  // ★ 单线程写，线程安全

executor.shutdown();  // 关闭线程池（不再接受新任务）
executor.awaitTermination(30, TimeUnit.SECONDS);  // 等待所有任务完成
⚠️ 注意：result 是 ArrayList（不是线程安全的！）， 所以合并结果时要加 synchronized (result)。
更好的方案：用 Future 代替 CountDownLatch（后面讲）。

模块四 CountDownLatch 原理（面试必考）

4.1 生活比喻

CountDownLatch = 倒计时门栓 比赛开始：10 个运动员准备 → 裁判喊"开始" → 所有人同时跑 → latch.await() = 主线程在门口等 → latch.countDown() = 每个运动员到达终点，计数器 -1 → 计数器到 0 → 门栓打开 → 主线程继续（开始汇总成绩） 关键：CountDownLatch 只能用一次（计数器到 0 后就废了）。 如果要再次使用，要新建一个 CountDownLatch。

4.2 源码核心（面试追问）

// CountDownLatch 的核心：AQS（AbstractQueuedSynchronizer）
public class CountDownLatch {
    private static final class Sync extends AbstractQueuedSynchronizer {
        Sync(int count) {
            setState(count);  // AQS 的 state 字段 = 计数器初始值
        }

        // await()：state != 0 时，线程进入等待队列
        protected int tryAcquireShared(int acquires) {
            return (getState() == 0) ? 1 : -1;  // state=0 → 获取成功（不阻塞）
        }

        // countDown()：state - 1，如果 state=0 就唤醒所有等待线程
        protected boolean tryReleaseShared(int releases) {
            for (;;) {
                int c = getState();
                if (c == 0) return false;  // 已经到 0 了，不用再释放
                int nextc = c - 1;
                if (compareAndSetState(c, nextc))  // CAS 原子操作
                    return nextc == 0;  // 如果到 0 了，唤醒所有等待线程
            }
        }
    }

    // 调用方代码
    public void await() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);  // 调用 AQS 的模板方法
    }
    public void countDown() {
        sync.releaseShared(1);  // 调用 AQS 的模板方法
    }
}
✅ 面试加分：说出 CountDownLatch 是基于 AQS（AbstractQueuedSynchronizer） 实现的， 内部用 state 字段表示计数器，用 CAS（Compare-And-Swap） 保证原子性。
这能体现你不仅"会用"，还知道"为什么能用"。

4.3 vs CyclicBarrier vs Semaphore（面试必背对比！）

工具作用是否可复用典型场景项目用的
CountDownLatch等待 N 个线程完成❌ 一次性等待所有分片查完✅ 用的是这个
CyclicBarrierN 个线程互相等待✅ 可复用分片之间需要交换数据时用❌ 没用
Semaphore限制同时访问的线程数✅ 可复用限制 DB 连接数时用❌ 没用
Q：为什么项目用 CountDownLatch 而不用 CyclicBarrier？

答：因为项目中"分片之间不需要交换数据"。

  • CountDownLatch：主线程等所有分片完成 → 汇总结果
  • CyclicBarrier：分片之间要互相等待，到了屏障点交换数据 → 分片协作
  • 项目里每个分片独立查自己的数据，不需要和其他分片交换 → 用 CountDownLatch 更简单

5.1 线程池篇

Q1：线程池的 7 个参数，你项目里怎么设置的？为什么？
ThreadPoolExecutor(
    corePoolSize = 8,     // CPU 核心数（IO + 计算混合场景）
    maximumPoolSize = 16,  // 最大线程数（实际上不会用到，因为队列满了就触发 CallerRunsPolicy）
    keepAliveTime = 60,   // 非核心线程空闲 60 秒后回收（节省资源）

    handler = new ThreadPoolExecutor.CallerRunsPolicy()  // 背压策略（防止 OOM）
)

为什么 corePoolSize = 8？ 导出是 IO + 计算混合场景，设 CPU 核心数。
如果是纯 IO 场景（比如只是转发请求），会设 CPU×2。

Q2：如果导出 1000 万数据，你的方案会出什么问题？怎么优化？
  • 问题 1：result 列表存全部数据 → OOM（内存存不下 1000 万条）
  • 问题 2：分片数太多 → 任务太多 → 调度开销大
  • 优化方案：改用 生产者-消费者模式（见下题）
Q3：更好的方案 —— 生产者-消费者模式
// 生产者：10 个线程查 DB，查到一批就放进队列
executor.submit(() -> {
    while (hasMoreData()) {
        List<RequestCountDO> batch = queryBatch();  // 查一批（比如 1000 条）
        blockingQueue.put(batch);  // 队列满了自动阻塞（背压！）
    }
});

// 消费者：1 个线程从队列取数据，写 Excel（流式写）
while (true) {
    List<RequestCountDO> batch = blockingQueue.poll(3, TimeUnit.SECONDS);
    if (batch == null) break;  // 所有生产者都完了
    excelWriter.write(batch);  // 流式写，不存全量数据
}

优点：不存全量数据到内存（用队列缓冲），支持任意大数据量导出。

5.2 CountDownLatch 篇

Q4：CountDownLatch 和 Future 的区别？项目中用哪个更好？
工具能否获取返回值能否取消任务项目适用性
CountDownLatch❌ 不能（只能等完成）❌ 不能✅ 可以用，但要自己合并结果（加锁）
Future✅ 能（future.get()）✅ 能（future.cancel()）✅ 更好！无需加锁，代码更简洁

用 Future 优化项目代码：

List<Future<List<RequestCountDO>>> futures = new ArrayList<>();
for (int i = 0; i < shardCount; i++) {
    futures.add(executor.submit(() -> queryShard(i)));  // 返回 Future
}
// 主线程收结果（无锁！）
for (Future<?> f : futures) {
    result.addAll(f.get());  // get() 自动等待，且无需 synchronized
}

5.3 FastExcel 篇

Q5：FastExcel 和 EasyExcel 的关系？
  • FastExcel 是 EasyExcel 的 fork 版本（原版 EasyExcel 停止维护）
  • API 完全兼容，可以无缝替换（改个 Maven 依赖就行）
  • 核心优化：SAX 模式解析（逐行读，不加载全量到内存）
  • 支持 流式写（excelWriter.write(batch) 可以多次调用，每次写一批）
Q6：EasyExcel/FastExcel 的 SAX 模式是什么？

传统 DOM 模式（不好）：

  • 把整个 Excel 文件加载到内存 → 大文件会 OOM
  • 适合小文件（几 MB）

SAX 模式（好！）：

  • 逐行解析，读到一行就回调一行 → 不加载全量到内存
  • 适合大文件（几百 MB）
  • FastExcel 读 Excel 默认用 SAX 模式（自动的，不用配置）

模块六 项目不足与优化方向（面试加分必说）

6.1 项目存在的不足（诚实说出来，加分！）

不足 1：synchronized (result) 合并结果会成为瓶颈。
优化方案：用 Future 代替 CountDownLatch， 每个分片返回自己的结果，主线程无锁合并（见 Q4）。
不足 2：如果导出 1000 万数据，result 列表会 OOM。
优化方案：改用"生产者-消费者模式"，不存全量数据到内存（见 Q3）。
不足 3：线程池没有复用，每次导出都创建新线程池。
优化方案：把线程池定义为 Spring Bean（@Bean），所有导出任务复用同一个线程池。
不足 4：没有做"导出进度"功能，用户不知道还要等多久。
优化方案：用 Redis 存进度（export:progress:{taskId}）， 前端通过 WebSocket 轮询或推送进度。

6.2 架构优化方向

优化 1：支持"异步导出"（提交导出任务后，返回 taskId， 用户可以去干别的，导出完了收到通知再来下载）。适合大数据量导出。
优化 2：限制同时导出的任务数（比如最多 3 个同时在导）， 防止多个用户同时导出把系统资源耗尽。用 Semaphore 实现。
优化 3：导出完成后自动上传到 OSS（对象存储）， 返回下载链接。适合分布式部署（多个实例共享文件）。
