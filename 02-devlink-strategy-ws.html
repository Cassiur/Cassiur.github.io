模块一 为什么要用策略模式？

1.1 不用设计模式的痛苦（坏味道）

项目要接入 7 个大模型：ChatGPT、讯飞星火、DeepSeek、百度千帆、阿里通义、豆包、智谱 GLM。 如果不用设计模式，你的代码会长这样： public ChatRecordsVo chat(Long user, String question, AISourceEnum aiType) { if (aiType == CHAT_GPT) { // 调 ChatGPT API 的逻辑（50行） } else if (aiType == XUNFEI) { // 调讯飞 API 的逻辑（50行） } else if (aiType == DEEP_SEEK) { // 调 DeepSeek API 的逻辑（50行） } ... // 一共 7 个分支，350行！ 问题 1：违反开闭原则 加第 8 个模型（比如 Kimi），就要改这个 if-else 方法 → 改旧代码 = 引入 Bug 的风险。 问题 2：代码膨胀 一个方法 350 行，读代码的人会疯掉。 问题 3：无法平行扩展 7 个模型的逻辑全耦合在一个方法里，改 ChatGPT 的逻辑可能不小心影响讯飞。

1.2 开闭原则（面试必背三句话）

对扩展开放：加新功能时，可以加新代码。
对修改关闭：加新功能时，尽量不修改已有代码。
怎么做到？用"抽象"隔离变化 —— 策略模式就是干这个的。

1.3 用了策略模式之后

加新模型 = 加一个新类，原来的代码一行都不用改！ // 第 8 个模型：KimiAiServiceImpl.java @Service public class KimiAiServiceImpl extends AbsChatService { @Override public AISourceEnum source() { return AISourceEnum.KIMI; } @Override protected ChatRecordsVo doAnswer(Long user, String question) { // 调 Kimi API 的逻辑（独立的 50 行） } } 原来的 chat() 方法？完全不用改。这就是"热插拔"。

模块二 策略模式三件套（源码逐行解读）

2.1 策略接口（ChatService.java）—— 定义"算法族"

文件位置：paicoding-service/src/main/java/com/github/paicoding/forum/service/chatai/service/ChatService.java

// 策略接口：定义了"聊天"这个行为的统一规范
// 所有大模型都必须实现这个接口 —— 这就是"抽象隔离变化"
public interface ChatService {

    // 每个 AI 实现类返回自己的类型标识
    // 工厂模式靠这个方法区分不同的策略
    AISourceEnum source();

    // 同步聊天：问完等到答完，一次性返回（适合简单的问答）
    // 返回值：ChatRecordsVo（包含 AI 的回答内容）
    ChatRecordsVo chat(Long user, String question);

    // 异步聊天：问完不等待，AI 生成一段推一段（流式，体验好）
    // 参数 consumer：一个回调函数，AI 每生成一段就调一次
    ChatRecordsVo asyncChat(Long user, String question,
                               Consumer<ChatRecordsVo> consumer);
}
✅ 接口的核心价值：调用方（ChatFacade）只依赖 ChatService 接口， 不依赖任何具体实现类。这就是"面向接口编程"，也是开闭原则的基础。

2.2 抽象模板（AbsChatService.java）—— 抽公共逻辑

文件位置：paicoding-service/src/main/java/com/github/paicoding/forum/service/chatai/service/AbsChatService.java

// 抽象模板：把"公共逻辑"抽到这里，子类只管"怎么调 AI API"
// 模板方法模式：定义一个算法骨架，把步骤延迟到子类实现
public abstract class AbsChatService implements ChatService {

    // === 公共逻辑：所有模型都要做的 ===

    // ① 检查使用次数是否超限（免费用户每天只能用 N 次）
    protected void checkUsageLimit(Long user) {
        if (queryUsedCnt(user) >= MAX_FREE_CNT) {
            throw new BusinessException("今日免费次数已用完，请明天再来或开通会员");
        }
    }

    // ② 使用次数 +1（写入 Redis 或 DB）
    protected void incrCnt(Long user) { ... }

    // ③ 查询已用次数
    protected int queryUsedCnt(Long user) { ... }

    // ④ 保存聊天记录到 DB（方便用户查看历史）
    protected void saveRecord(Long user, String question, ChatRecordsVo answer) { ... }

    // === 模板方法：定义算法骨架 ===
    // 这个方法实现了 ChatService.chat()，子类不用再实现
    @Override
    public ChatRecordsVo chat(Long user, String question) {
        // 步骤1：检查使用次数（公共）
        checkUsageLimit(user);

        // 步骤2：使用次数 +1（公共）
        incrCnt(user);

        // 步骤3：调子类的"真正聊天"方法（抽象，延迟到子类实现）
        //        这就是模板方法模式的核心：父类定义流程，子类实现细节
        ChatRecordsVo res = doAnswer(user, question);

        // 步骤4：保存聊天记录（公共）
        saveRecord(user, question, res);

        return res;
    }

    // === 抽象方法：子类必须实现 ===
    // 每个模型的 API 调用方式不同，所以交给子类实现
    protected abstract ChatRecordsVo doAnswer(Long user, String question);
}
✅ 模板方法模式：AbsChatService.chat() 定义了"算法骨架"（检查次数 → 调 AI → 保存记录）， 而 doAnswer() 是"延迟到子类实现的步骤"。这样所有模型的流程一致，但实现可以不同。

模块三 工厂模式：怎么拿到正确的策略？

3.1 为什么需要工厂？

策略模式解决了"算法可以替换"的问题，但谁来选哪个算法？ 不用工厂的写法： // 调用方自己判断用哪个策略 —— 还是要写 if-else！ ChatService service; if (aiType == CHAT_GPT) service = new ChatGptAiServiceImpl(); else if (aiType == XUNFEI) service = new XunFeiAiServiceImpl(); → 调用方还是依赖了具体实现类（new ChatGptAiServiceImpl()）， 没有做到"面向接口编程"。 → 加新模型，调用方也要改 → 还是违反开闭原则。

3.2 注册式工厂（Spring 魔法，项目精华！）

文件位置：paicoding-service/src/main/java/com/github/paicoding/forum/service/chatai/service/ChatServiceFactory.java

// 工厂类：负责"根据类型拿到正确的策略"
// @Component：注册为 Spring Bean，这样其他类可以 @Autowired 注入它
@Component
public class ChatServiceFactory {

    // 核心：策略注册表（Map<类型, 策略实例>）
    // 注意是 Map，不是 if-else！O(1) 查找，类型安全
    private final Map<AISourceEnum, ChatService> chatServiceMap;

    // ★★★ Spring 魔法：构造器注入 ★★★
    // Spring 会自动找到所有 ChatService 的实现类（7 个），
    // 把它们装进一个 List，传给这个构造器！
    // 这就是"注册式工厂"的核心原理。
    public ChatServiceFactory(List<ChatService> chatServiceList) {
        chatServiceMap = new HashMap<>();
        for (ChatService service : chatServiceList) {
            // 以 source() 的返回值为 key，注册到 Map
            // 比如：key=CHAT_GPT_3_5, value=ChatGptAiServiceImpl 实例
            chatServiceMap.put(service.source(), service);
        }
        // 构造器执行完后，chatServiceMap 里就有 7 个策略了
    }

    // 根据类型返回对应的策略（O(1) 查找）
    public ChatService getChatService(AISourceEnum aiSource) {
        return chatServiceMap.get(aiSource);
        // 如果 aiSource=CHAT_GPT_3_5 → 返回 ChatGptAiServiceImpl 实例
    }
}
✅ 这个工厂的精妙之处：
① 加新模型不用改工厂：新建一个 KimiAiServiceImpl，加 @Service， Spring 启动时会自动把它注入到 chatServiceList → 自动注册到 Map。
② O(1) 查找：用 Map 而不是 if-else，性能更好。
③ 类型安全：编译期就能发现类型错误，不像反射那样运行时才报错。

3.3 调用链路（一定要能画出来！）

前端发请求：POST /chat?aiType=CHAT_GPT&question=什么是策略模式 │ ▼ ChatFacade.chat(user, question, aiType) ← 门面模式（统一入口） │ ├─ ① 调工厂拿到策略：ChatServiceFactory.getChatService(CHAT_GPT) │ ↓ │ 返回 ChatGptAiServiceImpl 实例 │ └─ ② 调策略的 chat() 方法（多态！） ↓ 实际执行 AbsChatService.chat()（模板方法） ↓ doAnswer() ↓ 实际执行 ChatGptAiServiceImpl.doAnswer()
多态的魔力：chatService.chat() 这行代码， 在运行时根据实际类型执行不同的实现 —— 这就是"一个接口，多种实现"的含义。

模块四 面试八股文（策略模式篇）

Q1：策略模式和工厂模式各自解决什么问题？项目中怎么配合的？

答：

模式解决什么项目对应一句话记忆
策略模式把"算法族"封装成独立类，可以互相替换ChatService 接口 + 7 个实现类"换算法"
工厂模式把"创建对象的逻辑"集中管理ChatServiceFactory 根据类型返回对应策略"拿算法"
模板方法定义算法骨架，延迟步骤到子类AbsChatService.chat() 定义流程"算法骨架"

配合方式：工厂负责"创建/获取"（给调用方屏蔽创建细节），策略负责"行为替换"（让算法可以互换）。

Q2：除了 if-else，为什么不用反射来创建策略实例？

有人会想：直接 Class.forName("ChatGptAiServiceImpl").newInstance() 不行吗？

  • 反射性能差：每次都要加载类、检查权限、分配内存 → 比直接 new 慢 10~100 倍
  • 编译期发现不了错误：类名写错了，编译不会报错，运行时才抛 ClassNotFoundException
  • Spring 管理不了反射创建的对象：反射 newInstance() 出来的对象，Spring 不会给它注入依赖（比如 @Autowired 失效）
  • 工厂 + Map 的方案：O(1) 查找，类型安全，Spring 管理生命周期，支持 IDE 重构（重命名类时自动更新引用）
Q3：如果 7 个实现类有依赖冲突（比如用的 SDK 版本不同）怎么办？

这是策略模式在真实项目的挑战。解决方案：

  • 方案1：依赖隔离（复杂）：用独立的 ClassLoader 加载不同版本的 SDK。Java 的 SPI（Service Provider Interface）机制可以做到，但实现复杂，适合中间件开发。
  • 方案2：HTTP 调用（推荐！）：每个模型封装成独立的微服务（或 Lambda 函数），通过 HTTP/RPC 调用。彻底隔离依赖，还能独立部署、独立扩容。
  • 方案3：接受妥协：统一用一个兼容版本（项目当前做法）。适合模型 SDK 差异不大的场景。

面试加分回答："目前项目用的是方案3，因为 7 个模型的 SDK 依赖冲突不大。如果后续接入更多模型，会考虑方案2（拆成微服务），这样还能做到不同模型的独立扩容。"

模块五 WebSocket 流式推送（毫秒级响应体验）

5.1 先搞懂：为什么需要"流式"？

不用流式（坏体验）： AI 生成全部内容（假设 10 秒） → 一次性返回 → 用户盯着"加载中"转圈 10 秒 → 一次性看到全部内容。 （用户心理：怎么这么慢？是不是卡死了？） 用流式（好体验）： AI 生成第 1 个 token（50ms） → 立刻推送给用户 → 用户看到第 1 个字（感知到"在打字"） AI 生成第 2 个 token（50ms） → 立刻推送给用户 → 用户看到第 2 个字 ...（重复 200 次） AI 生成完毕 → 推送"结束"信号。 （用户心理：哇，它在打字！好快！） 总时间不变（都是 10 秒），但用户感知的"首屏时间"从 10 秒 → 50 毫秒！ 这就是"流式推送"的核心价值：不是更快，而是"感觉更快"。

5.2 项目架构：两段流式（SSE 收，WebSocket 发）

项目的流式架构很巧妙，分两段： 第一段（收流）：后端 → AI 大模型 后端调 AI API 时，AI 返回的是 EventSource（SSE，Server-Sent Events） → 后端每收到一个 token，就触发 onMsg() 回调 → 这是"收流"阶段 第二段（发流）：后端 → 前端浏览器 后端收到 token 后，通过 WebSocket 推送给前端 → 前端收到一个 token，就追加到页面上 → 这是"发流"阶段 为什么不用 SSE 直接推前端？ 因为项目用的是 STOMP over WebSocket（双向通信协议）， 前端不仅要收 AI 回复，还要发问题给后端 → 需要双向通信 → WebSocket 更合适。

5.3 WebSocket vs SSE vs HTTP 长轮询（面试必背对比）

维度HTTP 普通请求HTTP 长轮询SSE（Server-Sent Events）WebSocket
通信方向单向（客户端→服务端）单向（服务端→客户端，但每次要重新建立连接）单向（服务端→客户端）双向（全双工）
协议HTTPHTTPHTTP（text/event-stream）ws:// 或 wss://
连接开销每次请求都要建立连接（慢）每次响应后都要重新连接（慢）一次连接，长期保持一次握手，长期保持
适合场景普通 API 请求不推荐（已被 SSE/WS 取代）消息推送、进度通知聊天、游戏、协同编辑
项目用的❌❌❌（但后端收 AI 流用的是 SSE）✅ 前端和后端之间用的是 WebSocket
⚠️ 简历修正建议：你的简历写的是"后端结合 SSE 推送 Stream 流"， 但实际项目里，后端和前端之间用的是 WebSocket（STOMP 协议）， 后端和 AI 大模型之间用的是 SSE。面试时如果被追问"SSE 怎么实现的？"， 你应该回答两段流式的架构，而不是直接说"用了 SSE"。建议把简历改成： "后端结合 WebSocket（STOMP 协议）推送 Stream 流，实现前端毫秒级响应体验"。

5.4 STOMP 协议简单理解（类比普通话）

概念解释类比项目对应
WebSocket底层双向通信协议（定义怎么建立连接、怎么发数据帧）电话线（物理层）ws://localhost:8080/ws
STOMP在 WebSocket 之上的消息协议（定义帧格式：CONNECT、SEND、SUBSCRIBE...）打电话时的"普通话"（双方都懂的格式）@MessageMapping、@SendToUser
@MessageMapping服务端接收消息的入口（类似 @RequestMapping）电话铃响，接电话WsChatConfig.java
SimpMessagingTemplate服务端主动发消息给客户端的工具对着电话说话，对方能听到WsAnswerHelper.sendMsgToUser()

模块六 源码深挖：WebSocket 推送链路

6.1 前端怎么建立 WebSocket 连接？

前端代码（JavaScript）： // 用 SockJS（兼容不支持 WebSocket 的浏览器） const socket = new SockJS('/ws/chat'); const stompClient = Stomp.over(socket); // 连接成功后的回调 stompClient.connect({}, (frame) => { // 订阅服务端推送的消息（每个人的订阅地址不同，用 userId 区分） stompClient.subscribe('/user/queue/chat/reply', (message) => { // 收到 AI 的回复，追加到页面上 const data = JSON.parse(message.body); appendToPage(data.content); }); }); 关键：/user/queue/chat/reply 是"用户私人队列"，只有这个用户能收到。 服务端用 convertAndSendToUser() 发送时，Spring 会自动路由到正确的用户。

6.2 后端配置类：WsChatConfig.java

文件位置：paicoding-web/src/main/java/com/github/paicoding/forum/web/front/chat/ws/WsChatConfig.java

@Configuration
@EnableWebSocketMessageBroker  // ← 开启 WebSocket + STOMP 支持
public class WsChatConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        // 注册 STOMP 端点（前端连这个地址）
        // setAllowedOriginPatterns("*")：允许跨域（生产环境要限制域名！）
        // withSockJS()：降级支持（浏览器不支持 WebSocket 时用长轮询模拟）
        registry.addEndpoint("/ws/chat/{userId}/{aiType}")
                .setAllowedOriginPatterns("*")
                .withSockJS();
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        // 消息代理（负责把消息路由到正确的订阅地址）
        // enableSimpleBroker("/queue/")：前端订阅 "/user/queue/..." 时，由这个代理转发
        registry.enableSimpleBroker("/queue/");

        // 应用前缀（前端发消息时要带这个前缀）
        // 比如前端发消息到 "/app/chat/send"，会路由到 @MessageMapping("/chat/send")
        registry.setApplicationDestinationPrefixes("/app");
    }
}

6.3 推送消息的核心工具：WsAnswerHelper.java

文件位置：paicoding-web/src/main/java/com/github/paicoding/forum/web/front/chat/ws/WsAnswerHelper.java

// 这个类负责：把 AI 生成的 token，通过 WebSocket 推送给前端
@Component
public class WsAnswerHelper {

    // Spring 提供的 WebSocket 消息发送工具（核心！）
    @Autowired
    private SimpMessagingTemplate simpTemplate;

    // 推送消息给指定用户（AI 每生成一个 token，就调一次这个方法）
    public void sendMsgToUser(Long userId, AiChatItemVo chatRes,
                                  AiChatStatEnum stat, Consumer<AiChatStatEnum> consumer) {
        try {
            // convertAndSendToUser()：Spring 的"私信"方法
            // 参数1：用户名（这里用 userId 的字符串形式）
            // 参数2：订阅地址（前端订阅 "/user/queue/chat/reply" 才能收到）
            // 参数3：消息体（包含 AI 的 token 或"结束"信号）
            simpTemplate.convertAndSendToUser(
                    String.valueOf(userId),   // 用户名
                    "/queue/chat/reply",   // 订阅地址（Spring 会自动加 "/user" 前缀）
                    new WebSocketResponseDto(stat, chatRes)  // 消息体
            );
            // 调回调函数（通知调用方："这条消息已经发出去了"）
            consumer.accept(stat);
        } catch (Exception e) {
            log.error("WebSocket 推送消息失败！userId={}", userId, e);
        }
    }
}
✅ 面试常问：convertAndSendToUser() 是怎么做到"只发给指定用户"的？
答：Spring 在建立 WebSocket 连接时，会把 userId 和 WebSocketSession 绑定起来。 发消息时，根据 userId 找到对应的 Session，只往这个 Session 里写数据。

Q5：WebSocket 和 HTTP 的区别？为什么聊天要用 WebSocket？

答：

维度HTTP/HTTPSWebSocket
连接请求-响应，完事断开（短连接）一次握手，长期保持连接（长连接）
方向只能客户端发请求双向，服务端可以主动推
开销每次请求都带完整 Header（浪费带宽）连接建立后，数据帧很小（2~14 字节）
适用普通 API 请求、页面加载聊天、游戏、实时协同、股票行情
项目❌ 不适合（AI 回复要推给前端）✅ 用的就是这个

为什么聊天必须用 WebSocket？ 因为 AI 回复是"服务端主动推"的场景，HTTP 做不到（HTTP 只能客户端发请求）。

Q6：WebSocket 长连接怎么保活？心跳机制？

问题背景：

  • 中间代理（Nginx、负载均衡器）通常有"空闲超时"机制（比如 Nginx 默认 75 秒无数据就断开连接）
  • 网络抖动可能导致连接"假死"（TCP 连接还在，但数据发不出去）
  • 客户端网络切换（WiFi → 4G）会导致 IP 变化，旧连接失效

心跳机制（Ping/Pong）：

  • 客户端每隔 30 秒发一个 Ping 帧给服务端
  • 服务端收到 Ping 后，回一个 Pong 帧
  • 如果客户端超过 90 秒（3 个心跳周期）没收到 Pong → 认为连接断开 → 主动关闭连接并重连

项目里的实际情况：用的是 STOMP over WebSocket，STOMP 协议自带心跳帧（heart-beat header）。 在 WsChatConfig.java 里可以配置：registry.addEndpoint(...).setHeartbeats(new long[]{10000, 10000})（每 10 秒一次心跳）。

Q7：如果 WebSocket 连接断了，用户正在聊天的上下文怎么处理？
  • 重连机制：前端检测到断开 → 自动重新建立 WebSocket 连接（带重试次数限制，比如最多重试 5 次）
  • 上下文存储：聊天历史存在服务端 DB，重连后重新加载，不丢上下文
  • 消息补偿：断连期间 AI 生成的 token 暂存队列，重连后一次性推送（或丢弃，看产品需求）
  • 用户体验：前端显示"连接已断开，正在重连..."，而不是报错，减少用户焦虑
Q8：WebSocket 和 STOMP 的关系？项目中用的哪个？

答：WebSocket 是传输层协议（定义怎么建立连接、怎么发数据帧），STOMP 是消息层协议（定义消息格式：CONNECT、SEND、SUBSCRIBE、MESSAGE...）。

类比：WebSocket = 电话线，STOMP = 普通话（双方都懂的对话格式）。

项目中用的是 STOMP over WebSocket（在 WebSocket 连接之上跑 STOMP 协议）。 前端用 Stomp.over(sockJS) 建立连接，后端用 @MessageMapping 接收消息， 用 SimpMessagingTemplate.convertAndSendToUser() 发送消息。

模块八 项目不足与优化方向（面试加分必说）

8.1 项目存在的不足（诚实说出来，加分！）

不足 1：简历写的是"SSE 推送 Stream 流"，但实际项目里前端和后端之间用的是 WebSocket（STOMP 协议）， 后端和 AI 大模型之间用的是 SSE。两句话描述不准确，面试被追问会露馅。
修正方案：把简历改成"后端结合 WebSocket（STOMP 协议）推送 Stream 流，实现前端毫秒级响应体验"。
不足 2：没有做连接鉴权！当前 WsChatConfig.java 里 setAllowedOriginPatterns("*") 允许所有域访问， 且没有对 WebSocket 连接做用户身份验证。恶意用户可以伪造 userId，收到别人的 AI 回复。
优化方案：在 WebSocketConfigurer 里加一个 HandshakeInterceptor， 在 WebSocket 握手阶段就验证用户身份（比如校验 JWT Token），通过后才允许建立连接。
不足 3：没有做消息限流。恶意用户可以疯狂发消息给 AI，打垮服务。
优化方案：用 Redis + Lua 脚本实现滑动窗口限流，每个用户每秒最多发 5 条消息。
不足 4：7 个模型的 API Key 直接写在配置文件里（application.yml），没有加密。
优化方案：用 Spring Cloud Config + AES 加密，或者环境变量注入，避免敏感信息泄露。

8.2 架构优化方向

优化 1：把 7 个模型拆成独立的微服务（或 Serverless 函数），通过 HTTP/RPC 调用。 好处：① 依赖隔离（不同模型的 SDK 版本不冲突）；② 独立扩容（热点模型可以多开实例）；③ 独立部署（更新一个模型不影响其他模型）。
优化 2：加一个"模型健康检查"机制。定期 ping 每个模型的 API，如果超时，自动切换到备用模型（比如 ChatGPT 挂了，自动切到 DeepSeek）。
优化 3：用 Redis 缓存相同的提问（比如"什么是策略模式"），5 分钟内有人问同样的问题，直接返回缓存的回答，不用再调 AI API（省钱 + 更快）。
