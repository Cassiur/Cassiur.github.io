二、Caffeine 本地缓存：最快的缓存

2.1 为什么用 Caffeine（本地缓存）而不是全用 Redis？

  对比   Caffeine（本地缓存）   Redis（分布式缓存）  

  位置   在你的 Java 程序内存里   另一台服务器上  
  速度   读取约 0.1 微秒（几乎瞬时）   读取约 1 毫秒（要走网络）  
  能存多少   受你程序内存限制（一般几百 MB）   可以很大（几十 GB）  
  多实例共享   错误： 每个实例各自一份   正确： 所有实例共享同一份  

结论：
- 侧边栏数据（热门文章、公告） 指向  每个请求都要，但数据不常变  指向  用 Caffeine
- 文章点赞数（要实时、多实例共享） 指向  用 Redis

4.3 Pipeline vs 事务（MULTI/EXEC）

  对比   Pipeline   事务（MULTI/EXEC）  

  目的   减少 RTT，提升性能   保证多个命令原子执行  
  原子性   错误： 不保证（中间可能被其他命令插入）   正确： 保证（全部执行或全部不执行）  
  适用场景   批量读写，不要求原子性   转账（A 扣钱、B 加钱必须同时成功）  

项目里的选择：阅读数 +1 和点赞数 +1 不需要严格原子性（偶尔不一致完全可以接受），所以用 Pipeline 就够了。

六、面试高频追问：你能答上来吗？

Q1：如果 Redis 宕机了，计数数据会丢吗？

答：会丢 Redis 里的数据，但 MySQL 里还有。下次用户访问时，从 MySQL 重新加载（或者等凌晨定时任务恢复）。

Q2：Pipeline 一定能提升性能吗？

答：不一定。如果命令很少（比如只有 1-2 个），Pipeline 的效益不明显，反而增加了客户端代码的复杂度。一般 ≥3 个命令才值得用 Pipeline。

Q3：Caffeine 缓存的数据，服务重启了会丢吗？

答：会。Caffeine 是内存缓存，服务重启  指向  内存清空  指向  缓存全部丢失。但这完全可以接受（下次访问时重新加载到缓存就行了）。

Q4：你们项目里缓存的一致性是怎么保证的？

答：我们采用的是 最终一致性 方案：
1. 写操作：同时写 MySQL 和 Redis（Redis 用 hIncrBy 异步累加）
2. 读操作：优先读 Redis，Redis 没有再读 MySQL
3. 兜底：每天凌晨定时任务从 MySQL 重新统计，全量覆盖 Redis
