模块一 为什么需要消息队列？

1.1 没有消息队列的世界（痛点全景）

假设你点赞了一篇文章，系统需要同时做这些事：

❌ 同步处理的灾难链：

用户点【点赞】按钮
↓
① 往 user_foot 表写入点赞记录（20ms）
② 给文章作者发一条"XXX 赞了你的文章"（调用通知服务，50ms）
③ 更新用户活跃积分（写 Redis ZSet，5ms）
④ 更新文章点赞计数（写 Redis + DB，10ms）
⑤ 发站内信（10ms）
↓
总耗时 = 20+50+5+10+10 = 95ms → 用户要等 95ms 才能看到"已赞"

更可怕的是：如果通知服务挂了，点赞就一直转圈，用户只能用刷新大法。

1.2 有了消息队列之后

✅ 异步处理的丝滑体验：

用户点【点赞】按钮
↓
① 往 user_foot 表写入点赞记录（20ms）
② 发一条消息到 RabbitMQ（1ms）→ 立刻返回给用户
↓
用户 20ms 后就看到"已赞"了！

后台默默做（用户不关心）：
→ 消费者A 收到消息 → 发通知（50ms，用户不用等）
→ 消费者B 收到消息 → 更新积分（5ms）
→ 消费者C 收到消息 → 更新计数（10ms）
维度不用 MQ（同步）用 MQ（异步）
用户感知延迟95ms（慢）20ms（快 5 倍）
通知服务挂了的影响点赞功能全部失败用户无感知，消息堆积，服务恢复后继续处理
数据库压力瞬时写入压力大消费者可以控制消费速度（削峰）
代码耦合点赞逻辑里写死了"发通知"代码点赞只管发消息，通知逻辑完全独立
核心总结：消息队列解决两个问题——① 异步解耦（点赞不用等通知）+ ② 削峰填谷（海量点赞排队慢慢处理）。

1.3 RabbitMQ 在项目里的实际位置

paicoding-main 项目中的消息流转路径： 【用户点赞】 → UserFootServiceImpl.favorArticleComment() → DB: INSERT/UPDATE user_foot（点赞记录） → RabbitmqServiceImpl.publishMsg() → Exchange: pai_exchange_notify → RoutingKey: praise → Queue: pai_notify_praise → [消息在 RabbitMQ 里等着] → RabbitmqServiceImpl.consumerMsg() 的消费者 → NotifyServiceImpl.saveArticleNotify() → 作者收到点赞通知 【关键源码位置】 • 生产端：paicoding-service/.../notify/service/impl/RabbitmqServiceImpl.java • 消费端：同一个文件里的 consumerMsg() 方法 • 配置类：paicoding-core/.../config/RabbitmqProperties.java

模块二 RabbitMQ 核心概念（面试必背）

2.1 五个核心概念

概念中文作用项目对应
Producer生产者发消息的一方RabbitmqServiceImpl.publishMsg()
Exchange交换机接收生产者的消息，按规则转发给队列pai_exchange_notify（Direct 类型）
Queue队列存消息的地方，消费者从这里取pai_notify_praise
RoutingKey路由键Exchange 用它决定消息给哪个队列"praise"、"comment"
Consumer消费者收消息并处理的一方consumerMsg() 里的 DefaultConsumer

2.2 Exchange 的四种类型（面试高频）

① Direct Exchange（直连交换机）—— 项目用的就是这个！
规则：消息的 RoutingKey 和 Queue 的 BindingKey 完全匹配才路由。
项目例子：RoutingKey="praise" → 只有绑定了 "praise" 的队列能收到。

② Topic Exchange（主题交换机）
规则：支持通配符 *（一个单词）和 #（多个单词）。
例子：RoutingKey="user.log.info" 会匹配 BindingKey="user.#"。

③ Fanout Exchange（广播交换机）
规则：不管 RoutingKey，所有绑定到这个 Exchange 的队列都收到消息。
场景：发公告，所有在线用户都要收到。

④ Headers Exchange（头交换机）
规则：根据消息的 Header 属性匹配，而不是 RoutingKey。用得少。

模块三 生产端 Confirm 机制（消息不丢失第一关）

3.1 问题背景：消息发出去就一定能到 Broker 吗？

不一定！以下情况会导致消息丢失：

  • 网络抖动：消息根本没到 Broker，就在半路丢了
  • Broker 宕机：消息到了 Broker 的内存，但还没持久化，Broker 重启后丢失
  • Exchange 不存在：消息被 Broker 拒绝
  • 队列满了：新消息被直接丢弃

3.2 Confirm 机制原理（类比快递签收）

快递比喻RabbitMQ Confirm 机制说明
你把包裹给快递员，快递员说"好的收到"Broker 发 ACK消息安全到达 Broker，且已持久化到磁盘
快递员说"这个我们不收"Broker 发 NACK消息被 Broker 拒绝，需要生产端重发
快递员不说话（网络断了）超时没收到 ACK/NACK需要生产端重发
关键区别：Confirm 机制 vs 事务机制
txSelect/txCommit（AMQP 事务）：同步阻塞，每条消息都要等 Broker 确认，性能极差（降低 10 倍以上吞吐量）。
confirmSelect（Confirm 机制）：异步回调，发送消息后不阻塞，Broker 确认后回调 ConfirmListener，性能接近不用事务。

3.3 源码逐行解读：publishMsg()（第 47-161 行）

// 文件：paicoding-service/src/main/java/.../notify/service/impl/RabbitmqServiceImpl.java

public void publishMsg(String exchange, BuiltinExchangeType exchangeType,
                        String routingKey, String message) {
    RabbitmqConnection rabbitmqConnection = null;
    Channel channel = null;
    try {
        // ① 从连接池拿一个连接（连接池复用，避免频繁建连）
        rabbitmqConnection = RabbitmqConnectionPool.getConnection();
        Connection connection = rabbitmqConnection.getConnection();
        channel = connection.createChannel();

        // ② 声明交换机（durable=true 表示交换机持久化）
        //     如果交换机已存在，这行不会重复创建
        channel.exchangeDeclare(exchange, exchangeType, true, false, null);

        // ③ ★ 开启 Confirm 机制（这行之后，Broker 会对每条消息发 ACK/NACK）
        channel.confirmSelect();

        // ④ ★ 注册 Confirm 回调（异步！Broker 确认后会调这里）
        channel.addConfirmListener(
            // 成功回调（Broker 说：消息我已经收到并持久化了）
            (deliveryTag, multiple) -> {
                log.info("消息已到达 Broker，deliveryTag={}", deliveryTag);
                // multiple=true 表示所有 <= deliveryTag 的消息都确认了（批量确认）
            },
            // 失败回调（Broker 说：消息处理出错了，你需要重发）
            (deliveryTag, multiple) -> {
                log.warn("消息被 Broker 拒绝，需要重发！deliveryTag={}", deliveryTag);
                // ★ 这里应该加重发逻辑！但项目里没写（后面讲项目不足）
            }
        );

        // ⑤ 发送消息
        //    注意：这里没设置 MessageProperties.PERSISTENT_TEXT_PLAIN
        //    → 消息本身没持久化！（项目可优化点）
        channel.basicPublish(exchange, routingKey, null, message.getBytes(StandardCharsets.UTF_8));

        // ⑥ ★ 同步等待 Confirm（最多等 3 秒）
        //    如果 3 秒内没收到 ACK → 抛 TimeoutException
        //    这行是"可靠性"和"性能"的取舍：宁可慢一点，也要保证消息不丢失
        channel.waitForConfirmsOrDie(3000);

    } catch (Exception e) {
        // ⑦ 捕获异常：超时 / Broker 拒绝 / 网络错误
        log.error("消息发送失败，需要重发策略", e);
        // ★ 项目不足：这里只是打日志，没有重发逻辑！
        //    面试时可以说优化方案：把失败的消息写入本地 DB，定时任务扫描重发
    } finally {
        // ⑧ 归还连接给连接池（必须！否则连接耗尽）
        if (channel != null && channel.isOpen()) {
            try { channel.close(); } catch (Exception e) { log.error("channel close error", e); }
        }
        if (rabbitmqConnection != null) {
            RabbitmqConnectionPool.returnConnection(rabbitmqConnection);
        }
    }
}

3.4 面试追问：waitForConfirmsOrDie 是同步还是异步？

Q：addConfirmListener 是异步回调，为什么还要 waitForConfirmsOrDie？

答：这是项目在"可靠性"和"性能"之间做的取舍。

  • addConfirmListener 是纯异步的 → 发完消息就返回，不管 Broker 有没有收到。适合"允许少量丢失"的场景（如日志收集）。
  • waitForConfirmsOrDie(3000) 会阻塞当前线程，直到 Broker 确认收到（或超时 3 秒）。适合"消息不能丢"的场景（如支付、通知）。
  • 项目选了阻塞等待 → 宁可慢一点，也要保证点赞通知不丢失。
  • 更好的方案：异步 Confirm + 重发队列。Confirm 超时后，把消息写入 message_retry 表，定时任务扫描并重发。
❌ 项目不足（面试要主动说）：
① catch 里只是打日志，没有重发逻辑！
② basicPublish 没设置 MessageProperties.PERSISTENT_TEXT_PLAIN，消息只在 Broker 内存里，Broker 宕机重启后消息丢失。
③ waitForConfirmsOrDie 超时后直接抛异常，没有重试。应该用 waitForConfirms(timeout)（返回 boolean）然后自己控制重试。

模块四 消费端手动 Ack（消息不丢失第二关）

4.1 自动 Ack vs 手动 Ack（面试必考对比）

方式行为优点缺点 / 风险项目选择
自动 Ack
autoAck=true		Broker 一把消息发给消费者，立刻认为"已消费"，直接从队列删除性能好，代码简单如果消费者收到消息但还没处理完就挂了 → 消息永久丢失❌ 不用
手动 Ack
autoAck=false		消费者显式调用 basicAck() 后，Broker 才删除消息消息不丢失，消费者挂了消息会重新投递如果忘了写 basicAck → 消息一直堆积（但不丢）✅ 项目用的这个
✅ 为什么项目选手动 Ack？ 点赞通知不能丢。如果自动 Ack，消费者刚收到消息就挂了，作者永远收不到点赞通知，用户会投诉。

4.2 三个 Ack 方法的区别（面试必背）

方法作用消息是否重新入队典型场景
basicAck(deliveryTag, multiple)确认消息已成功处理❌ 不重新入队（消息被删除）正常处理成功
basicNack(deliveryTag, multiple, requeue)告知 Broker 处理失败✅ 可以重新入队（requeue=true）处理失败，希望重新分配
basicReject(deliveryTag, requeue)拒绝消息（单次拒绝）✅ 可以重新入队和 basicNack 类似，但不能批量拒绝

4.3 源码逐行解读：consumerMsg()（第 80-163 行）

// 文件：paicoding-service/src/main/java/.../notify/service/impl/RabbitmqServiceImpl.java

public void consumerMsg(String exchange, String queueName, String routingKey) {
    RabbitmqConnection rabbitmqConnection = RabbitmqConnectionPool.getConnection();
    Connection connection = rabbitmqConnection.getConnection();
    Channel channel = connection.createChannel();

    try {
        // ① 声明交换机（和生产者对应）
        channel.exchangeDeclare(exchange, BuiltinExchangeType.DIRECT, true, false, null);

        // ② ★ 声明死信交换机（兜底方案！后面详细讲）
        channel.exchangeDeclare(EXCHANGE_NAME_DLX, BuiltinExchangeType.DIRECT, true);
        //     死信队列
        channel.queueDeclare(DLX_QUEUE_NAME_PRAISE, true, false, false, null);
        //     绑定死信交换机和死信队列
        channel.queueBind(DLX_QUEUE_NAME_PRAISE, EXCHANGE_NAME_DLX, DLX_QUEUE_KEY_PRAISE);

        // ③ ★ 主队列绑定死信交换机（关键配置！）
        Map args = new HashMap<>();
        //     当消息变成死信时，转发到这个交换机
        args.put("x-dead-letter-exchange", EXCHANGE_NAME_DLX);
        //     死信的 RoutingKey（可以和运动队列的不同）
        args.put("x-dead-letter-routing-key", DLX_QUEUE_KEY_PRAISE);
        //     消息 TTL：30 秒没被消费 → 变死信（防止消费者挂了消息永远在队列里）
        args.put("x-message-ttl", 30000);

        channel.queueDeclare(queueName, true, false, false, args);
        channel.queueBind(queueName, exchange, routingKey);

        // ④ ★ 创建消费者（手动 Ack 模式）
        Consumer consumer = new DefaultConsumer(channel) {
            @Override
            public void handleDelivery(String consumerTag, Envelope envelope,
                                     AMQP.BasicProperties properties, byte[] body) {
                String message = new String(body, StandardCharsets.UTF_8);
                try {
                    // ⑤ 处理消息：保存通知记录
                    notifyService.saveArticleNotify(
                        JsonUtil.toObj(message, UserFootDO.class),
                        NotifyTypeEnum.PRAISE
                    );
                    // ⑥ ★ 手动 Ack（告诉 Broker："我处理完了，可以删消息了"）
                    channel.basicAck(envelope.getDeliveryTag(), false);
                    //     ↑ 如果这里不调用，Broker 会认为消息没处理完，不会删除
                    //       如果消费者挂了，消息会重新投递给其他消费者（至少处理一次）
                } catch (Exception e) {
                    log.error("消息处理失败", e);
                    // ⑦ ★ 处理失败 → 拒绝消息且不重新入队 → 消息进死信队列
                    //     requeue=false：不重新入队，直接变死信
                    channel.basicNack(envelope.getDeliveryTag(), false, false);
                    //     ↑ 如果 requeue=true，消息会重新入队，可能无限循环重试！
                }
            }
        };

        // ⑧ ★ autoAck = false → 关闭自动 Ack
        channel.basicConsume(queueName, false, consumer);

    } catch (Exception e) {
        log.error("consumer init error", e);
        RabbitmqConnectionPool.returnConnection(rabbitmqConnection);
    }
}

4.4 面试追问：basicNack 的 requeue 参数怎么选？

Q：处理失败时，requeue 应该设 true 还是 false？

答：分情况。

  • requeue=true：消息重新入队，Broker 会把它分配给另一个消费者（或者同一个消费者）。
    适用：临时故障（如 DB 连接暂时断了），重试可能成功。
    风险：如果是代码 Bug（如空指针），消息会无限重试，打死消费者。
  • requeue=false：消息不重新入队，直接变成死信，进死信队列。
    适用：永久性故障（如消息格式错误、代码 Bug）。
    好处：不会无限重试，死信队列里的消息可以后续人工排查或修复 Bug 后重发。

项目里用的是 requeue=false，这是更安全的做法。因为如果是代码 Bug，重试 N 次也成功不了，不如进死信队列让人工处理。

模块五 死信队列（DLQ）兜底方案

5.1 什么是死信（Dead Letter）？

死信 = 无法被正常消费的消息。RabbitMQ 会把死信转发到"死信交换机（DLX）"，再路由到"死信队列（DLQ）"。

5.2 变成死信的三种情况（面试必背）

1消息被拒绝且不重新入队
channel.basicReject(tag, false) 或 channel.basicNack(tag, false, false)
项目里的用法：catch 到异常 → basicNack(tag, false, false) → 消息进死信队列。
2消息 TTL 过期
项目里设置了 x-message-ttl=30000（30 秒）。如果消息 30 秒没被消费（比如消费者挂了），就变死信。
为什么设 30 秒？ 防止消费者挂了，消息永远堆积在队列里，把队列撑爆。
3队列满了，新消息进来时最老的消息被踢出
如果队列设置了 x-max-length，队列满了之后新消息会踢掉最老的消息，被踢掉的变死信。

5.3 项目中的死信架构图

正常队列 "pai_notify_praise" │ │ 消息处理失败（basicNack, requeue=false） │ OR 消息 30 秒没被消费（TTL 过期） ▼ 死信交换机 "dlx_notify" │ │ RoutingKey = "dlx_praise" ▼ 死信队列 "dlx_queue_praise" │ └─ 这里存了所有死信，供人工排查或批量重处理
💡 死信队列的核心价值：即使消费端出 Bug，消息也不会丢失，而是进死信队列。后续可以修复 Bug 后手动重发，或者写个死信消费者自动重试（指数退避策略）。

5.4 面试追问：死信队列里的消息怎么处理？

Q：死信队列里的消息，你们项目是怎么处理的？

答：项目里只是把死信转发到了死信队列，但没有写死信消费者的代码。这是可以优化的点。

三种处理方式：

  • ① 人工排查：运维同学定期查看死信队列，找出失败原因，修复 Bug 后手动重发。
  • ② 自动重试消费者：写一个死信队列的消费者，收到死信后等待一段时间（指数退避：1s → 2s → 4s → 8s），然后再重新投递到主队列。但要注意防止无限重试（设一个最大重试次数，超过就进"最终死信队列"）。
  • ③ 告警：死信队列有消息时，发告警邮件/短信给运维同学，人工介入。
❌ 项目不足（面试要主动说）：死信队列只有"存"的能力，没有"消费重试"的能力。面试时可以说优化方案：写个死信消费者，用指数退避策略自动重试，超过最大重试次数后发告警人工介入。

模块六 消息幂等性（防止重复消费）

6.1 问题：同一条消息可能被消费两次

场景：
① 消费者处理了消息，调了 basicAck()
② 但网络抖动，Ack 包没到 Broker
③ Broker 认为消息没处理完 → 重新投递给另一个消费者
④ 同一条消息被处理了两次！

后果：作者收到两条一模一样的"XXX 赞了你的文章"通知。

6.2 解决方案（面试必背）

方案原理优点缺点项目适用性
① 数据库唯一约束消息体里带 messageId（UUID），消费时先 SELECT ... WHERE msg_id = ?，如果已存在就跳过简单可靠需要 DB 唯一索引✅ 项目可以用
② Redis SetSADD msg_id，返回 0 表示已处理过性能好Redis 挂了会丢幂等记录✅ 项目可以用
③ 业务本身幂等看业务逻辑是否天然幂等无需额外存储不是所有业务都天然幂等✅ 项目正好符合！

6.3 项目天然幂等的原因

// 项目的点赞记录存在 user_foot 表
// 这张表有唯一索引：uk_user_document(user_id, document_id, document_type)
//
// 同一个人对同一篇文章：
//   - 第一次点赞：INSERT 成功
//   - 消息重复消费，又调了一次 saveArticleNotify()
//     → 但 notify 表也应该有幂等设计（根据 messageId 去重）
//     → 或者：通知内容相同，用户收到两条一模一样的通知（不幂等，需要优化）

// 面试加分回答：
// "我们的 user_foot 表有唯一索引，所以点赞记录不会重复。
//  但通知表如果没有幂等设计，可能会发两条相同的通知。
//  优化方案：通知表加 msg_id 字段 + 唯一索引，防止重复插入。"
面试加分回答模板：
"我们的项目里，user_foot 表有唯一索引，所以点赞记录本身不会重复。
但通知表如果没有幂等设计，消息重复消费时可能会插入两条相同的通知记录。
优化方案：① 通知表加 msg_id 字段 + 唯一索引；② 消费时先查 SELECT id FROM notify WHERE msg_id = ?，如果已存在就跳过。"

7.1 RabbitMQ 基础篇

Q1：RabbitMQ 和其他消息队列（Kafka、RocketMQ）的区别？
维度RabbitMQKafkaRocketMQ

面试回答："我们的场景是点赞通知，消息量不大（万级），但对可靠性要求高，所以选了 RabbitMQ。如果是日志收集，会用 Kafka。"

Q2：什么是消息堆积？怎么解决？

消息堆积 = 生产者发消息的速度 >> 消费者处理消息的速度，导致消息在队列里越堆越多。

解决方案：

  • ① 增加消费者数量（最简单）：多开几个消费者实例，RabbitMQ 会自动负载均衡。
  • ② 限流生产端：如果消费者已经处理不过来了，生产端应该停止发送（背压）。
  • ③ 优化消费端性能：消费端处理一条消息要 50ms？能不能优化到 10ms？
  • ④ 设置队列最大长度：x-max-length，队列满了之后新消息会踢掉最老的消息（或变死信），防止队列无限增长。

7.2 消息可靠性篇

Q3：怎么保证消息不丢失？（追问率 99%）

答：从三个层面保证。

① 生产端 Confirm 机制（消息从生产者到 Broker 不丢）

  • 开启 channel.confirmSelect()
  • 注册 addConfirmListener 回调，处理 ACK/NACK
  • waitForConfirmsOrDie(3000) 同步等待确认（或异步 + 重发表）

② 消息持久化（Broker 宕机不丢）

  • 交换机持久化：exchangeDeclare(..., durable=true, ...)
  • 队列持久化：queueDeclare(..., durable=true, ...)
  • 消息持久化：basicPublish(..., MessageProperties.PERSISTENT_TEXT_PLAIN, ...)
    ⚠️ 项目里没设置这个！面试要主动说这是可优化点。

③ 消费端手动 Ack（消费者挂了不丢）

  • basicConsume(..., autoAck=false, ...) → 关闭自动 Ack
  • 处理完成后 basicAck(...) → 才告诉 Broker 删除消息
  • 如果处理到一半消费者挂了 → 没有 Ack → Broker 认为没处理完 → 重新投递（至少处理一次）
Q4：Confirm 机制和事务机制的区别？为什么不用事务？
方式原理性能可靠性
AMQP 事务
txSelect/txCommit		同步阻塞，每条消息都要等 Broker 确认非常差（降低 10 倍以上吞吐量）高

项目选了 Confirm 机制，兼顾可靠性和性能。

7.3 高级篇

Q5：RabbitMQ 的镜像队列是什么？解决什么问题？

问题背景：默认情况下，RabbitMQ 的队列只存在于一个节点上。如果这个节点宕机了，队列里的消息就丢了。

镜像队列（Mirror Queue）：把队列镜像到多个节点上，主节点挂了，镜像节点自动接管，消息不丢失。

面试回答："我们生产环境配了镜像队列，保证高可用。但如果同时写入量很大，镜像队列的性能会比普通队列差一些（要同步到多个节点）。"

Q6：消息的 TTL 和死信队列，你们项目是怎么用的？

项目里设置了 x-message-ttl=30000（30 秒）。意思是：如果消息 30 秒没被消费（比如消费者挂了），就变死信，进死信队列。

为什么设 30 秒？ 防止消费者挂了，消息永远堆积在队列里，把队列撑爆。30 秒后变死信，进死信队列，后续可以排查。

面试加分："我们还可以针对不同的业务设置不同的 TTL。比如点赞通知可以 30 秒，但支付通知要 5 分钟（因为支付可能延迟）。"

模块八 项目不足与优化方向（面试加分必说）

8.1 项目存在的不足（诚实说出来，加分！）

不足 1：生产端 catch 里只是打日志，没有重发逻辑。
优化方案：捕获异常后，把失败的消息写入 message_retry 表（包含 msg_id、body、retry_count、next_retry_time），定时任务每秒扫描 next_retry_time <= NOW() 的记录，重新发送，最多重试 3 次，超过后发告警人工介入。
不足 2：basicPublish 没设置 MessageProperties.PERSISTENT_TEXT_PLAIN，消息只在 Broker 内存里。
优化方案：channel.basicPublish(exchange, routingKey, MessageProperties.PERSISTENT_TEXT_PLAIN, message.getBytes())
不足 3：死信队列只有"存"的能力，没有"消费重试"的能力。
优化方案：写个死信消费者，用指数退避策略（1s → 2s → 4s → 8s）自动重试，超过最大重试次数后发告警人工介入。
不足 4：通知表没有幂等设计，消息重复消费时会插两条相同的通知记录。
优化方案：通知表加 msg_id 字段 + 唯一索引，消费时先查重。

8.2 架构优化方向

优化 1：生产端异步 Confirm + 重发表（而不是 waitForConfirmsOrDie 同步等）。这样发送消息的性能更高，重发逻辑也更可靠。
优化 2：消费者用线程池并发处理（但要注意顺序问题）。比如一个用户的点赞通知，不能并发处理，否则可能通知顺序错乱。可以用 userId 取模，相同 userId 的消息分给同一个线程处理。
优化 3：监控 + 告警。队列长度超过阈值、消费者挂了、死信队列有消息，都要发告警。可以用 RabbitMQ 的 rabbitmq-prometheus 插件 + Grafana 做监控看板。
